# 校园乐跑项目开发文档

## 1. 项目定位

本项目面向高校学生运动打卡场景，构建一个包含微信小程序学生端、Spring Boot 后端、网页管理端的完整系统。当前目标是交付一个可真实上线的单校内部 MVP，而不是仅用于演示的原型。

## 2. 工程化约束

- 为保证单校上线版本稳定可运维，后端采用模块化单体，而不是直接拆分微服务
- 保留按领域拆分的包结构与接口边界，便于后续继续演进
- 学生端采用微信登录主流程，并通过学号密码完成首次绑定
- 跑步轨迹采用前端采集 + 后端校验的模式，兼顾真实采集与实现成本
- 配置按 `dev / prod` 分离，开发种子数据只在 `dev` 环境生效

## 3. 总体架构

### 3.1 架构组成

- `miniapp/`：原生微信小程序学生端
- `backend/`：Spring Boot 3.x REST API 服务
- `admin-web/`：Vite + Vue 管理后台
- `docs/`：开发与部署文档
- `docker-compose.yml`：本地开发环境编排

### 3.2 核心模块

- 认证权限模块：账号鉴权、JWT 签发、角色校验、微信绑定
- 用户模块：学生信息、个人主页、体能统计
- 跑步模块：开始跑步、轨迹采集、暂停结束、记录查询、有效性校验
- 激励模块：任务、勋章、等级、积分
- 社交模块：动态、点赞、评论、跑团消息
- 活动模块：活动发布、报名、教程推荐
- 运营管理模块：学生导入、审核、日志、数据看板

## 4. 技术选型

### 4.1 后端

- JDK 17
- Spring Boot 3.5.x
- Spring Web / Validation / Security / Actuator
- Spring Data JPA
- MySQL 8
- Redis 7
- Flyway
- springdoc-openapi

### 4.2 管理端

- Vue 3
- Vite
- Vue Router
- Pinia
- Axios
- Element Plus

### 4.3 小程序端

- 微信小程序原生框架
- WXML / WXSS / JavaScript
- 封装 REST API 请求层
- 微信登录与定位能力

## 5. 数据库设计原则

- 所有核心业务对象使用独立表存储，保持查询与治理清晰
- 跑步轨迹快照允许以 JSON 字符串方式存储，接口层按摘要与详情拆分
- 通过审计日志记录后台关键操作，满足基础运营审计需求
- 排行榜结果在 Redis 中缓存，同时保留 MySQL 作为事实数据源
- 学号与微信 `openid` 建立唯一约束，保证绑定关系稳定

## 6. 功能实现策略

### 6.1 学生端

- 登录：微信登录，未绑定时进入学号绑定流程
- 首页：轮播图、个人周目标、热门动态、推荐活动、排行榜摘要
- 跑步：支持开始、暂停、继续、结束；展示距离、时长、配速、卡路里
- 社交：动态发布、点赞、评论、跑团消息
- 我的：个人资料、历史跑步记录、勋章墙、任务进度

### 6.2 管理端

- 控制台：用户数、跑步里程、审核待办、活动统计
- 学生管理：名单导入、详情、禁用状态、微信绑定重置
- 内容审核：待审队列、风险标签、推荐置顶
- 活动管理：活动发布与教程管理
- 勋章等级：勋章配置、等级门槛配置
- 审计日志：后台操作查询

## 7. 非功能要求

- 支持本地一键启动开发环境
- 支持生产环境通过环境变量注入敏感配置
- 前后端分离，接口统一以 `/api` 暴露
- 核心业务对象具备参数校验与空值安全处理
- 关键链路具备基础审计日志、健康检查与限流能力

## 8. 里程碑

- M1：完成文档与项目骨架
- M2：完成后端核心实体、鉴权、数据库迁移
- M3：完成小程序端核心页面与真实跑步链路
- M4：完成管理端核心页面与学生运营能力
- M5：完成联调、部署配置与生产化治理
