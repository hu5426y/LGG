# 基于微信小程序的校园跑步打卡与社交系统的设计与实现

## 摘要
在高校体育管理不断向数字化、精细化发展的背景下，传统校园跑主要依赖人工组织、线下签到和分散统计，普遍存在记录不连续、监督成本高、学生参与积极性不足以及社交互动弱等问题。为提高校园跑的过程管理能力和学生持续参与度，本文结合微信小程序轻量化、即开即用的特征，设计并实现了一套基于微信小程序的校园跑步打卡与社交系统。系统围绕“跑步记录、每日打卡、成长激励、跑团社交、活动运营、后台管理”六个方面展开，实现了学生端、服务端和管理端三端协同。

本文在需求分析的基础上，采用前后端分离的总体思路，以微信小程序作为学生使用入口，以 Spring Boot 构建后端业务服务，以 Vue 3 和 Element Plus 构建管理后台，并使用 MySQL 持久化业务数据，Redis 缓存排行榜结果，Flyway 管理数据库版本演进。系统在功能上实现了账号密码登录、首页运营内容展示、跑步开始与结束、轨迹采集、暂停恢复、历史记录查看、跑步有效性校验、跑步计划选择、每日打卡、积分等级、任务成长、徽章授予、排行榜展示、动态发布、点赞评论、举报审核、跑团消息、活动报名、教程查看、学生导入、内容审核、徽章维护、活动发布、数据看板和操作日志等功能。

在实现过程中，本文重点研究了校园跑场景下的关键问题。针对跑步记录真实性不足的问题，系统在跑步结束阶段引入最小里程、最小时长、最少轨迹点数以及最大速度阈值等多维校验规则，并结合小程序端定位轨迹、加速度采样与云端原始轨迹补充机制，提升记录可信度。针对学生参与积极性不高的问题，系统通过积分、等级、任务、徽章、连续打卡与跑步计划等设计形成持续激励链路。针对校园跑平台缺乏互动的问题，系统增加了动态、评论、点赞、举报、跑团和活动报名等社交化能力，使校园跑由单一打卡行为转变为具有陪伴感和组织性的日常运动行为。

从测试结果看，系统能够较稳定地完成主要业务流程，核心模块之间的数据流转正确，后台可对学生、内容和活动进行有效管理，排行榜与统计看板能够满足基础运营分析需求。实践表明，该系统具有较好的可用性、扩展性和落地价值，可为高校校园跑数字化建设提供一套可复用的实现思路。未来仍可在地图纠偏、异常跑步识别、消息实时推送、规则个性化配置以及多维统计分析等方面继续完善。

**关键词：** 微信小程序；校园跑；跑步打卡；社交系统；Spring Boot；Redis

## Abstract
With the continuous digital transformation of campus sports management, traditional campus running still relies heavily on offline check-in, manual organization, and fragmented data collection. Such a mode often leads to weak supervision, poor continuity of records, limited student engagement, and insufficient social interaction. To address these problems, this paper designs and implements a campus running check-in and social system based on WeChat Mini Program. The system is built around six key aspects: running record management, daily check-in, growth incentives, squad-based social interaction, activity operation, and administrative management, forming a collaborative architecture of student client, backend service, and web-based administration console.

Based on requirement analysis, the system adopts a front-end and back-end separated architecture. The student side is implemented through WeChat Mini Program, the backend is developed with Spring Boot, and the administration console is built using Vue 3 and Element Plus. MySQL is used for business data persistence, Redis is used for ranking cache, and Flyway is used for database schema version control. The system supports password-based login, home-page content display, run start and finish, route collection, pause and resume, history query, run validity verification, running plan selection, daily check-in, points and level growth, task progress, badge granting, leaderboard display, social posts, likes, comments, reports, squad messaging, activity registration, tutorial browsing, student import, content moderation, badge maintenance, activity publishing, metrics dashboard, and audit logs.

This paper focuses on several key issues in the campus running scenario. To improve the authenticity of running data, the system introduces a multi-rule validation mechanism including minimum distance, minimum duration, minimum route point count, and maximum speed threshold. Route collection on the mini program side, accelerometer sampling, and optional cloud-side raw route backup are combined to enhance record credibility. To improve student motivation, the system connects points, levels, tasks, badges, streak check-ins, and running plans into a persistent incentive loop. To strengthen interaction, social features such as posts, comments, likes, reports, squads, and activity registration are introduced, transforming campus running from a single check-in task into a more participatory and organized activity.

Testing results show that the system can complete the major business workflows in a stable manner. Data flow between core modules is correct, the administration side can effectively manage students, content, and activities, and the ranking and metrics panels satisfy basic operational analysis needs. The practical implementation demonstrates good usability, extensibility, and deployment value, providing a feasible reference for digital campus running platforms. Future improvements may focus on map correction, abnormal run detection, real-time messaging, configurable rule engines, and richer statistical analysis.

**Key words:** WeChat Mini Program; campus running; check-in; social platform; Spring Boot; Redis

## 第一章 绪论

### 1.1 研究背景
近年来，高校体育教育与学生体质健康管理越来越强调过程监督和长期坚持。校园跑作为许多高校普遍采用的课外锻炼形式，具有组织成本低、覆盖面广和实施灵活等特点，已经成为提升大学生日常运动频次的重要抓手。但在实际运行中，传统校园跑大多存在几个明显问题。第一，信息收集方式分散。学生的跑步情况往往依赖人工登记、截图证明或第三方工具零散上报，体育部门难以形成统一的数据视图。第二，监督成本较高。跑步是否真实完成、路线是否有效、是否存在代跑等问题，仅靠人工抽查难以长期维持。第三，激励机制较弱。多数校园跑系统把重点放在“完成”而不是“坚持”，学生容易把跑步当作被动任务。第四，社交属性不足。校园跑本身具有陪伴、竞争和分享的天然场景，但很多系统只记录结果，没有形成互动氛围。

与此同时，微信小程序在校园应用中的普及为校园跑数字化建设提供了新的实施路径。相较于独立 App，小程序具有无需安装、进入门槛低、易于传播、便于维护等特点，更符合学生群体高频、碎片化的使用习惯。学生可以直接在微信内完成登录、查看计划、开始跑步、参与活动和浏览动态，不必在多个平台之间频繁切换。对于学校而言，小程序前端发布和运维成本较低，也更利于后续推广。

在技术层面，移动端定位、地图展示、传感器采样、服务端鉴权、缓存和统计分析等能力已经较为成熟，为校园跑平台构建提供了可行基础。基于这些条件，本文结合本地实际项目，围绕校园跑核心业务构建一套面向学生和管理人员的完整系统，使校园跑从单一记录工具逐步扩展为集运动管理、成长激励、社交互动和运营分析于一体的数字平台。

### 1.2 研究目的及意义
本文的研究目的不是简单实现一个“能够记录跑步数据”的小程序，而是围绕高校校园跑场景，设计一套从学生使用、数据落库、规则校验到后台运营的完整业务链路。系统目标主要体现在三个方面。

第一，提升校园跑记录的规范性与可管理性。通过统一的账号体系、跑步记录流程、有效性校验规则和后台管理界面，可以减少线下重复统计和人工核验带来的工作量，使校园跑数据更加结构化、可追溯。

第二，增强学生参与校园跑的持续动力。将跑步任务与连续打卡、积分等级、徽章奖励、跑步计划和排行榜结合起来，可以让学生在完成基础跑步任务之外，感受到过程反馈和阶段成长，从而提高主动参与意愿。

第三，构建更加完整的校园跑社交与运营场景。校园跑不是孤立行为，学生往往有交流经验、约跑、参加活动和分享成果的需求。通过动态发布、评论点赞、跑团消息和活动报名等功能，系统可以把运动行为和校园社交连接起来，形成更有黏性的使用环境。

本研究的意义主要体现在实践和方法两个层面。实践层面上，本文项目提供了一套可运行的校园跑平台原型，对于高校体育部门、学院辅导员和学生社群管理都具有一定参考价值。方法层面上，本文将微信小程序、Spring Boot、Redis、Flyway 和 Vue 管理后台组织为一套清晰的三端架构，对类似的校园运动数字平台建设具有借鉴意义。

### 1.3 国内外研究现状

#### 1.3.1 校园跑数字化管理研究现状
国内高校近几年在数字校园建设中逐渐引入线上运动打卡、跑步签到和体锻积分系统，校园跑已经从线下组织转向线上记录与管理结合的模式。已有研究和实践普遍表明，校园跑平台的价值不仅在于记录学生跑步数据，更在于通过统一平台实现过程监督、结果统计和行为激励。很多现有平台已经能够实现基础里程记录、跑步次数统计和成绩展示，但在真实轨迹校验、持续激励和社交互动方面仍存在不足。一些平台重视制度约束，却忽视学生体验，导致学生对系统的接受度不高；另一些平台强调活动宣传，却缺乏足够扎实的数据结构和后台能力，难以支撑长期运营。

从现有问题来看，校园跑系统的核心难点主要集中在三个方向。一是记录真实性。单次跑步是否真实、轨迹是否连续、速度是否异常，是校园跑平台必须解决的问题。二是激励持续性。仅用一次性排名或结果通报很难让学生形成长期运动习惯，需要将日常打卡、任务目标和成长反馈结合起来。三是运营支撑能力。系统不仅服务学生端，也要支持学校进行学生管理、活动发布、内容审核和数据分析，否则很难在真实校园环境中长期应用。

#### 1.3.2 微信小程序与运动社交应用研究现状
微信小程序由于入口轻、使用便捷、传播成本低，已经成为校园类应用的重要载体。相关研究表明，小程序在教学服务、校园生活、健康管理等场景中均具有较高接受度。对于运动类应用而言，小程序能较好承载轻量化打卡、活动通知、社交互动和数据展示等功能，特别适合学生群体高频使用的业务场景。

在运动社交产品方向，国内外已有不少成熟应用采用了排行榜、动态分享、挑战任务、勋章体系和社群互动等手段来提升用户黏性。相关研究普遍认为，运动社交平台的持续使用意愿往往并不只由运动记录本身决定，还与同伴监督、成就反馈、社交比较和参与氛围密切相关。从这个角度看，校园跑系统如果只强调“打卡”而没有社交和激励，很难获得较好的长期使用效果。

不过，面向高校校园场景的系统与商业健身平台仍然有明显不同。高校系统更强调学生身份管理、统一规则、后台监督和低成本落地，而商业运动产品更关注用户自由度和个性体验。因此，校园跑平台需要在“管理可控”与“使用友好”之间找到平衡，既要满足体育管理需求，也要尽量减少学生对系统的排斥感。

### 1.4 研究内容
本文研究内容围绕本地项目的真实实现展开，主要包括以下几个部分。

首先，完成校园跑业务场景的需求梳理与系统目标定义。本文从学生端和管理端两个视角分析系统所需支撑的业务流程，明确系统应覆盖登录认证、跑步记录、打卡成长、社交互动、活动管理和数据统计等核心模块。

最后，对系统进行测试分析与总结。本文结合代码中的单元测试和系统级功能验证，对关键业务流程的正确性、可用性和稳定性进行说明，并分析当前系统仍然存在的不足和未来优化方向。

## 第二章 开发环境与相关技术

### 2.1 开发语言与工具
本系统采用多端协同开发方式，不同部分所使用的开发语言和工具有所区别。后端部分采用 Java 17 作为主要开发语言，使用 Spring Boot 3.5.0 构建业务服务。小程序端采用微信小程序原生页面结构配合 JavaScript 编写交互逻辑，并通过微信开发者工具进行调试和真机验证。管理后台使用 Vue 3、Vite 和 Element Plus 搭建，辅以 Axios 完成接口访问。数据库使用 MySQL 8.0，缓存使用 Redis 7，接口调试和文档展示通过 Swagger UI 完成。

在开发工具层面，项目提供了 Docker Compose 本地联调方案，可一键启动 MySQL、Redis、后端服务和管理后台，从而降低环境搭建复杂度。数据库结构和演示数据通过 Flyway 自动迁移，既保证了版本可追踪，也方便在不同环境中重复部署。项目同时包含多份本地联调说明文档和 Windows 启动说明，对实际开发协作具有较强的支持作用。

### 2.2 系统开发平台
从系统运行平台看，本文项目由学生端、管理端和服务端三部分组成。

学生端运行在微信小程序环境中，主要承担首页展示、跑步采集、动态互动、活动浏览和个人成长中心等功能。小程序端直接面向最终学生用户，对交互流畅性和使用便捷性要求较高。

服务端运行在 Spring Boot 环境中，承担统一的业务逻辑处理、接口返回、规则校验、数据库访问和缓存控制。后端通过 REST 风格接口向小程序和管理后台提供能力，是整个系统的数据中枢和规则执行中心。

管理端运行在浏览器中，采用单页面应用方式构建，主要面向管理员提供学生管理、内容审核、活动发布、徽章维护、统计分析和日志审计等功能。相较于学生端，管理后台更强调信息密度、操作效率和数据展示能力。

### 2.3 系统框架

#### 2.3.1 小程序端框架
小程序端采用微信小程序页面架构组织应用，入口文件通过 `app.json` 配置页面路径、窗口样式、定位权限和底部导航栏。系统一共包含登录、首页、跑步、动态、活动、个人中心、排行榜、成长中心、活动详情、教程详情、动态详情和跑步结果等页面，形成较为完整的学生端闭环。

小程序端使用自定义请求封装统一管理接口调用，并自动附带登录令牌。为适应不同联调方式，请求层同时支持直连 HTTP 接口和微信云托管容器两种模式。当目标接口不适合真机访问时，请求层还会给出明确的提示信息，减少开发调试期间的定位成本。跑步模块结合微信定位能力、地图展示能力和加速度传感器采样，在移动端持续采集轨迹点，支持开始、暂停、恢复、结束和本地恢复等流程。

#### 2.3.2 后端框架
后端采用 Spring Boot 作为基础开发框架，使用 Spring Web 对外暴露 REST 接口，使用 Spring Data JPA 处理数据持久化，使用 Spring Security 和 JWT 完成鉴权控制，使用 Spring Validation 进行参数校验，使用 Actuator 提供健康检查，使用 Springdoc OpenAPI 输出接口文档。

项目按业务域划分包结构，包括认证、首页、跑步、签到、跑步计划、成长激励、社交、活动、用户和后台管理等模块。这样的组织方式能够使控制器、服务类、实体类和数据访问层之间保持清晰边界，便于后续维护和扩展。

#### 2.3.3 管理后台框架
管理后台采用 Vue 3 组合式 API 组织页面逻辑，使用 Vite 作为构建工具，使用 Element Plus 提供表格、表单、弹窗、日期选择器等常用界面组件。针对统计分析场景，系统引入 ECharts 进行趋势图和分布图绘制。

后台在路由层设置了登录拦截逻辑，对非公开页面进行统一守卫；在请求层统一注入令牌，并对 401、403 等未授权响应自动清理本地会话并跳转登录页，从而保证后台使用过程中的权限一致性。

### 2.4 系统总体架构
本系统采用典型的前后端分离架构。学生通过微信小程序发起请求，管理员通过浏览器访问后台页面，二者都通过统一的 `/api` 接口与 Spring Boot 服务端通信。服务端负责业务处理、权限校验、数据存储和缓存管理，并将最终结果返回给前端。

从部署视角看，系统可通过 Docker Compose 以容器方式快速启动。MySQL 提供业务数据存储，Redis 提供排行榜缓存，后端容器负责业务接口，管理后台容器以 Nginx 静态资源方式提供页面访问。这样的结构适合本地联调、课程设计演示和后续扩展部署。

### 2.5 关键技术简介

#### 2.5.1 Spring Security 与 JWT
系统所有非公开接口都需要登录后访问。后端通过 Spring Security 配置无状态会话模式，将 `/api/auth/**`、`/api/public/**`、健康检查和 Swagger 文档等接口设为公开访问，其余接口都要求令牌认证。用户登录成功后，系统根据用户信息生成 JWT，其中包含用户名、用户编号和角色信息，默认有效期为 120 分钟。

这种方式的优点在于服务端不再依赖传统 Session 保存登录状态，既降低了横向扩展难度，也便于小程序端和后台端共享同一套认证机制。对于角色敏感接口，如后台管理接口，系统在控制器层使用方法级权限控制，确保只有管理员能够访问。

#### 2.5.2 MySQL 与 Flyway
MySQL 是系统的核心持久化数据库，用于存储用户、跑步记录、活动、动态、评论、徽章、打卡和计划等业务数据。为了保证数据库结构在不同环境中的一致性，项目使用 Flyway 维护迁移脚本。当前迁移脚本至少包含三个阶段：初始表结构创建、运行记录字段扩展、跑步计划与统计模块补充。这样做能够将数据库演进过程纳入版本管理，避免“库表结构与代码不一致”的常见问题。

#### 2.5.3 Redis
Redis 在本系统中主要承担排行榜缓存任务。排行榜属于查询频繁、更新相对可控的场景，如果每次都直接从数据库中排序获取，将增加后端和数据库压力。系统将总里程排行榜按固定键值缓存到 Redis 中，并设置 30 分钟过期时间。当用户完成一次有效跑步后，后端主动清除排行榜缓存，下次请求再重新生成最新排名结果，从而在性能和一致性之间取得平衡。

#### 2.5.4 微信定位与地图能力
跑步记录的核心在于轨迹采集。小程序端在跑步开始后申请定位权限，并调用定位更新能力持续接收用户当前位置。系统将采集到的经纬度、时间戳、精度和速度等信息封装为轨迹点，并实时计算距离、时长、配速、步数估算和卡路里消耗。首页和结果页还会根据历史轨迹构建地图预览，使学生能够直观查看最近一次跑步路线。

#### 2.5.5 传感器采样与原始轨迹补充
为了提升跑步记录可信度，小程序端在真实跑步模式下会开启加速度计监听，并根据采样结果生成运动置信度摘要。系统还提供可选的云端原始轨迹保存能力，将轨迹点按批次写入云数据库或本地降级存储，供调试和校验使用。这种做法虽然不会直接替代业务主记录，但能为后续异常分析和数据核验留下更细粒度的依据。

#### 2.5.6 Docker Compose
为降低环境搭建成本，项目使用 Docker Compose 管理本地依赖与服务启动。开发者只需执行统一脚本，即可同时拉起 MySQL、Redis、后端和管理后台。对于课程设计和团队协作而言，这种方式能显著减少“环境配置不一致”带来的问题，提高系统复现实验结果的稳定性。

#### 2.5.7 ECharts
管理后台的数据看板采用 ECharts 绘制趋势图和分布图。系统把近 14 天的跑步统计数据组织为日期维度序列，展示活跃跑者数、当日里程和计划完成数等趋势变化；同时根据连续打卡情况划分区间，生成打卡分布图。图形化展示不仅便于管理员理解数据，也使后台从单纯的事务处理工具扩展为基础运营分析平台。

### 2.6 本章小结
本章介绍了系统所采用的开发语言、运行平台、框架选择和关键技术，并说明了这些技术在项目中的具体作用。通过这些技术组合，系统具备了三端协同、快速部署、统一鉴权、数据缓存和统计分析的基础条件。

## 第三章 系统可行性与需求分析

### 3.1 系统目标
本系统的总体目标是构建一套适用于高校校园跑场景的轻量化数字平台，使学生能够通过微信小程序完成日常跑步记录与成长互动，使管理员能够通过网页端完成数据管理和运营支持。围绕这一总体目标，系统进一步细化为以下几个子目标。

其一，形成完整、可信的跑步记录链路。系统不仅要支持学生开始和结束跑步，更要对跑步记录进行基本有效性判断，减少明显异常数据进入结果统计。

其二，形成以打卡和计划为核心的持续锻炼机制。系统需要把单次跑步结果转化为连续打卡、计划推进和成长反馈，使学生在长期使用过程中建立稳定习惯。

其三，形成围绕运动的互动场景。系统要提供动态、评论、点赞、跑团和活动等功能，增强学生之间的交流感，避免校园跑平台沦为单纯的任务提交工具。

其四，形成可操作的后台管理能力。系统必须能够支持学生名单导入、学生状态更新、动态审核、活动发布、徽章配置、运营统计和日志追踪，满足管理使用需求。

### 3.2 系统可行性分析

#### 3.2.1 经济可行性
本系统的开发成本主要体现在开发时间、人力投入和基础环境资源三个方面。项目所使用的主要框架和中间件均为开源技术，包括 Spring Boot、Vue 3、MySQL、Redis、Flyway 和 Docker Compose，不需要额外购买昂贵的商业软件许可。学生端基于微信小程序开发，减少了跨平台原生 App 的重复开发成本。对于高校课程设计或中小规模部署而言，整体投入较低，具备较强的经济可行性。

#### 3.2.2 技术可行性
系统所依赖的关键技术均较成熟。微信小程序具备稳定的页面能力、网络能力和定位能力；Spring Boot 在 Web 服务和数据处理中应用广泛；MySQL 适合处理中等规模结构化业务数据；Redis 适合解决排行榜等高频查询问题；Vue 3 和 Element Plus 能够较快构建后台管理界面。项目源代码中已经完整实现了三端结构、数据库迁移、测试用例和本地联调脚本，说明整体技术方案能够落地。

#### 3.2.3 操作可行性
从学生端使用角度看，小程序的学习成本较低。用户只需完成账号密码登录，就可以通过底部导航进入首页、跑步、动态、活动和个人中心，不需要安装额外应用，也不需要学习复杂的操作流程。跑步过程中的开始、暂停、恢复、结束等动作符合常见运动应用的交互习惯。对于管理员而言，后台页面按模块进行划分，学生管理、内容审核、活动发布和数据看板功能入口清晰，适合日常操作。

#### 3.2.4 安全与管理可行性
系统在安全层面采用了统一的 JWT 鉴权机制，并在登录接口加入了基于 IP 的限流控制，默认 60 秒窗口内最多允许 12 次登录尝试，有助于减少暴力尝试风险。后台接口使用角色控制，学生无法访问管理员功能。动态模块加入了敏感词初筛与举报后复审逻辑，后台还能查看审计日志。以上设计说明系统具备基础的安全与管理可行性。

### 3.3 业务角色分析
本系统涉及两类核心角色。

第一类是学生用户。学生是系统的主要使用者，负责登录、查看首页内容、开始跑步、结束跑步、查看历史记录、完成每日打卡、参与跑步计划、获取积分与徽章、浏览排行榜、发布动态、加入跑团、查看活动和报名活动等。学生端强调的是便捷、高频和反馈及时。

第二类是管理员。管理员负责维护系统秩序和运营内容，包括查看学生信息、导入学生名单、更新学生状态、审核动态内容、配置徽章规则、发布活动和教程、查看统计看板以及浏览操作日志。管理端强调的是集中处理、信息完整和操作可追溯。

### 3.4 功能需求分析

#### 3.4.1 首页聚合展示需求
首页承担学生进入系统后的第一视图功能，需要集中展示当前值得关注的内容。根据项目实现，首页应包括轮播横幅、热门动态、推荐活动、跑步教程、排行榜和个人概览信息，同时还要显示当天打卡状态、当前跑步计划和最近一次跑步预览。首页的需求重点不是单一模块深挖，而是将分散信息以聚合方式组织出来，帮助学生快速进入后续操作。

#### 3.4.2 跑步记录需求
跑步记录模块是系统核心。该模块需要支持开始跑步、实时采集轨迹点、暂停跑步、恢复跑步、结束跑步和放弃未完成记录等操作。结束跑步后，系统还需要保存距离、时长、配速、卡路里、步数估算、轨迹点数量、设备信息、运动置信度和原始轨迹快照等数据，并将其纳入个人历史记录。

考虑到校园跑场景对记录真实性有基本要求，系统还需要在结束时对最小时长、最小里程、最少轨迹点数和最大速度阈值进行校验，避免显著异常记录被直接计入统计结果。

#### 3.4.3 打卡成长需求
学生持续跑步不能只依赖一次性结果，系统需要提供成长型功能来增强使用黏性。因此系统需要支持每日打卡、连续打卡天数计算、跑步计划模板选择、计划日进度推进、任务进度积累、积分发放、等级提升和徽章授予等功能。这样可以把一次跑步行为转化为多个成长反馈节点，增强学生对系统的持续关注。

#### 3.4.4 社交互动需求
校园跑具有天然的社交属性。系统需要允许学生发布跑步动态、浏览他人动态、点赞、评论和举报不良内容；同时还要支持跑团列表、加入跑团、退出跑团、查看成员和发送群消息等功能。社交模块既为学生提供分享和约跑的空间，也为后续形成积极的校园运动氛围创造条件。

#### 3.4.5 活动运营需求
系统需要支持管理员发布跑步活动和训练教程，学生能够在活动页浏览活动列表、查看详情和报名参加。教程内容则为学生提供训练建议、热身拉伸和跑步知识补充。活动和教程模块是校园跑平台由“工具型系统”向“运营型平台”延伸的重要部分。

#### 3.4.6 后台管理需求
后台管理需求包括学生数据管理、动态审核、徽章维护、活动和教程发布、统计看板展示以及审计日志查看。其中学生导入功能需兼容 `.csv` 和 `.xlsx`，内容审核需支持通过、拒绝、下架和置顶等处理动作，统计看板需展示活跃用户、里程趋势、连续打卡分布等基础运营信息。

### 3.5 非功能需求分析
除了业务功能外，系统还需要满足若干非功能需求。首先是可维护性。项目需要具备清晰的模块划分和数据库迁移机制，便于后续迭代。其次是可部署性。本项目通过 Docker Compose 和本地脚本降低了部署复杂度。再次是响应性能。排行榜采用 Redis 缓存，热点数据查询效率较高。最后是可测试性。后端提供了多组单元测试，覆盖登录、跑步校验、签到、计划、社交和后台服务等关键模块。

### 3.6 本章小结
本章从系统目标、可行性、角色和需求四个层面分析了项目边界，明确了校园跑平台需要解决的问题和需要实现的功能。后续章节将在这些分析结果的基础上给出系统总体设计与详细实现。

## 第四章 系统总体设计

### 4.1 设计原则
系统总体设计遵循以下几个原则。第一，围绕真实业务设计，而非单纯堆叠功能。校园跑的主线是跑步记录与成长反馈，因此其他模块都围绕这一主线服务。第二，前后端职责清晰。小程序与后台负责展示和交互，核心规则由服务端统一执行。第三，数据结构可扩展。系统在初始版本中已经为计划、打卡、徽章、跑团和统计等功能留出了独立数据表，便于后续演进。第四，重要操作可追踪。后台关键动作写入审计日志，保证管理过程具有可回溯性。

### 4.2 系统架构设计
从逻辑架构上看，系统分为表现层、业务层、数据层和支撑层。

表现层由微信小程序和 Vue 管理后台组成。小程序主要负责学生端页面交互和终端能力调用；后台负责管理员操作与统计图表展示。

业务层位于 Spring Boot 服务中，按照领域模块拆分为认证、首页、跑步、签到、跑步计划、成长激励、社交、活动、用户和后台管理等服务。每个服务对外提供接口，对内完成规则处理和数据组织。

数据层由 MySQL 和 Redis 构成。MySQL 保存核心业务数据，Redis 缓存排行榜结果。数据库演进由 Flyway 管理。

支撑层包括 Docker Compose、Swagger 文档、Actuator 健康检查和本地联调脚本，为开发、部署和运维提供辅助。

### 4.3 系统功能模块设计
根据业务实现，系统可划分为以下模块。

第一，认证与用户模块。负责学生和管理员登录、令牌签发、个人信息读取和会话状态维护。

第二，首页聚合模块。负责汇总横幅、热门动态、活动、教程、排行榜和个人概览信息。

第三，跑步记录模块。负责跑步状态流转、轨迹采集、结果提交、历史查询和记录有效性校验。

第四，打卡与成长模块。负责每日打卡、连续天数统计、跑步计划模板、计划推进、积分等级、任务进度和徽章授予。

第五，社交模块。负责动态、评论、点赞、举报、跑团加入退出、成员查看和群消息发送。

第六，活动运营模块。负责活动发布、活动报名和教程内容管理。

第七，后台管理模块。负责学生导入、状态维护、内容审核、徽章配置、数据统计和日志审计。

### 4.4 数据库设计

#### 4.4.1 数据库设计思路
数据库设计以“用户”为核心实体展开，围绕学生日常跑步行为向外扩展。`sys_user` 表负责用户基础信息和累计成长数据；`run_session` 表负责每次跑步记录；`daily_checkin` 表负责日打卡和连续签到信息；`user_run_plan` 与 `user_run_plan_day` 表负责个人跑步计划；`feed_post`、`post_comment`、`post_like` 和 `post_report` 表负责动态互动；`club`、`club_member` 和 `club_message` 表负责跑团组织；`activity` 与 `activity_registration` 表负责活动报名；`badge`、`user_badge`、`challenge_task`、`user_task_progress` 和 `level_rule` 表负责成长激励；`run_daily_stats` 表负责统计结果沉淀；`audit_log` 表负责后台关键操作追踪。

这种设计方式保证了不同业务之间边界清晰，既能支撑当前功能，也有利于后续扩展。

#### 4.4.2 主要数据表设计

**1. 用户表 `sys_user`**

用户表保存账号、密码、姓名、学号、学院、班级、头像、角色、状态、累计里程、累计时长、积分和等级值等信息。该表既服务于登录鉴权，也承载成长累计结果。

**2. 跑步记录表 `run_session`**

跑步记录表是系统核心业务表，主要字段包括跑步状态、开始时间、暂停时间、结束时间、距离、时长、平均配速、卡路里、步数估算、轨迹快照、轨迹点数量、设备型号、平台、客户端版本、云端原始会话编号和运动置信度等。该表能够完整描述一次跑步过程和结果。

**3. 打卡表 `daily_checkin`**

打卡表用于记录用户在某一天是否完成有效打卡，并维护连续天数字段。表中还关联了触发打卡的跑步记录和对应的计划日，便于追溯打卡来源。

**4. 跑步计划表 `run_plan_template`、`user_run_plan`、`user_run_plan_day`**

模板表定义计划类型、持续天数、目标里程和目标次数；用户计划表保存学生当前选中的计划；计划日表则拆分出每日训练目标及完成情况。通过这组三表，系统可以支持不同类型的训练计划和逐日推进。

**5. 动态社交表**

`feed_post` 表保存动态正文、图片、点赞数、评论数、举报数、风险标签和审核状态；`post_comment` 负责评论；`post_like` 负责点赞；`post_report` 负责举报。这些表共同完成学生动态交流及后台审核链路。

**6. 跑团表**

`club` 表定义跑团信息，`club_member` 表维护成员关系和角色，`club_message` 表保存群消息。通过这三张表可以实现加入跑团、退出跑团和群消息查看等功能。

**7. 活动表**

`activity` 表存储活动标题、地点、时间、报名截止时间、封面、描述、人数上限和状态；`activity_registration` 表记录报名关系。系统通过人数限制和报名截止时间控制报名行为。

**8. 成长激励表**

`badge` 表定义徽章授予规则，`user_badge` 记录用户已获得徽章；`challenge_task` 定义每日、每周或赛季任务，`user_task_progress` 保存任务完成进度；`level_rule` 定义积分与等级的映射关系。这些表支撑了积分、等级、任务和徽章体系。

**9. 统计与日志表**

`run_daily_stats` 表按日期沉淀活跃用户数、里程、时长、配速、打卡人数、计划完成数、活跃跑团和消息数量；`audit_log` 表记录后台关键操作，用于追踪和审计。

#### 4.4.3 数据库关系说明
从关系上看，用户与跑步记录、打卡记录、动态、评论、点赞、举报、跑团成员关系、活动报名关系、徽章和任务进度都属于一对多关系。跑步计划则是用户与计划模板之间的桥接关系，每个用户计划又包含多个计划日。动态和活动都与用户形成行为关系。通过这些关系设计，系统能够完整记录学生的运动行为、成长状态和社交互动轨迹。

### 4.5 接口与安全设计
后端接口统一以 `/api` 为前缀，对外分为公开接口和认证接口两类。公开接口包括登录、首页公共数据、健康检查和文档接口；其余业务接口都需要携带 JWT。小程序和后台前端在请求封装中自动附带 `Authorization: Bearer <token>`，服务端在过滤器中解析令牌并恢复用户身份。

对于后台接口，控制器通过 `@PreAuthorize("hasRole('ADMIN')")` 进行角色限制，学生无法访问后台资源。登录接口增加了限流控制，减少恶意频繁尝试风险。对于前端，收到 401 或 403 响应后会自动清理本地会话状态并跳转登录页，确保身份失效后的使用状态一致。

### 4.6 本章小结
本章从总体架构、模块划分、数据库设计和接口安全设计四个方面给出了系统总体方案。通过这一设计，系统能够支撑校园跑场景下的数据记录、成长激励、社交互动和后台管理需求。

## 第五章 系统详细设计与实现

### 5.1 登录认证模块设计与实现
系统采用账号密码登录方式，支持学生和管理员共用统一认证接口。学生端登录页内置了开发演示账号填充能力，方便联调时快速体验；后台则通过独立登录页进行管理员登录。用户提交账号和密码后，请求发送至 `/api/auth/login`，后端首先根据请求来源 IP 进行限流检查，随后查询用户信息、校验账号状态是否为启用状态，并对密码进行比对。校验成功后，系统生成 JWT 并返回给前端。

小程序端会将令牌和用户基础信息存入本地缓存，后续请求统一从缓存中读取令牌附加到请求头中。后台前端的 Axios 实例也通过拦截器自动注入令牌。该实现方式有两个直接优点：一是避免了多端重复实现鉴权逻辑；二是后端不需要维护服务器会话，简化了扩展。

从异常处理角度看，系统对于账号被禁用、密码错误和请求过于频繁等场景都返回明确提示，有利于用户理解失败原因。对于未登录或令牌过期的访问，后端统一返回未授权信息，前端再执行清理和跳转逻辑。

### 5.2 首页模块设计与实现
首页页面承载平台运营信息和个人运动概览。小程序端进入首页后，会并发请求公共首页数据、当前用户资料、成长概览、今日打卡和当前计划信息，必要时还会根据最近一次跑步编号拉取跑步详情，用于构建路线预览。这一设计避免了将所有数据塞入单一接口，提高了模块边界清晰度，也方便后续独立扩展。

首页数据主要包括五部分。第一部分是轮播横幅，用于展示当前推荐任务、重点活动或热点内容。第二部分是热门动态，用于展示校园跑中具有传播性的学生内容。第三部分是推荐活动和教程，为学生提供活动报名与训练知识入口。第四部分是排行榜，用于形成适度竞争氛围。第五部分是个人概览，包括用户名、累计里程、积分、等级、当日打卡状态、当前计划和最近一次跑步轨迹预览。

如果系统检测到用户最近有完成的跑步记录，首页会加载详细轨迹并绘制预览地图；如果没有历史记录，则尝试获取当前位置作为地图中心，使首页视觉呈现更完整。该实现增强了首页与跑步主业务之间的联动关系。

### 5.3 跑步记录模块设计与实现

#### 5.3.1 跑步状态流转设计
跑步记录模块采用状态机方式组织业务流程，核心状态包括运行中、暂停中和已完成。学生点击开始跑步后，小程序端先检查定位权限和当前定位可用性，再向服务端发起 `/api/runs/start` 请求创建新的跑步会话。服务端会先检查当前用户是否已存在未结束的跑步记录，若存在则拒绝重复开始。

开始成功后，小程序端进入采集状态，启动计时器、定位监听和加速度监听，并持续更新页面上的时长、里程、配速、步数和卡路里等指标。学生在跑步中可点击暂停，系统会将跑步状态更新为暂停，同时停止实时增长逻辑；继续跑步时再恢复状态。若学生决定放弃本次记录，也可以发起丢弃操作，将未完成记录从服务端删除。

#### 5.3.2 轨迹采集与本地恢复设计
跑步过程中，小程序端通过 `wx.startLocationUpdate` 和 `wx.onLocationChange` 持续接收定位更新，并将经纬度、时间戳、精度和速度等信息转换为轨迹点。每当获得新轨迹点后，系统会基于相邻点的球面距离公式计算新增距离，并据此更新累计里程。

考虑到小程序可能被切到后台、页面可能被关闭或意外中断，系统将当前跑步状态、轨迹点、开始时间、暂停时长、云端会话编号和若干衍生指标同步写入本地缓存。在用户重新进入跑步页时，系统优先读取本地缓存恢复现场；如果本地无缓存，则尝试调用 `/api/runs/current` 接口恢复未完成的服务端记录。这样的设计显著提升了实际使用中的容错能力。

#### 5.3.3 云端原始轨迹补充设计
除业务主记录外，小程序端还支持将原始轨迹点按批次写入云数据库或本地降级存储。系统首先创建原始轨迹会话，再在跑步过程中分批追加轨迹点，最终在结束跑步时补充摘要信息。这部分数据不直接参与业务展示，但可作为调试、追踪和异常分析依据。

在课程设计场景中，这种设计具有两个价值。其一，可以作为业务记录之外的辅助校验来源。其二，可以为未来扩展轨迹回放、路径分析和离线补传提供基础。

#### 5.3.4 跑步有效性校验设计
校园跑系统要避免显著异常记录直接入库，因此服务端在跑步结束时引入了多项校验规则。首先，跑步时长不能低于 300 秒，距离不能低于 0.5 千米，轨迹点不能少于 5 个。其次，平均速度不能高于 25 千米每小时，以排除明显不符合跑步行为特征的记录。再次，系统会遍历轨迹点序列，检查时间戳是否严格递增，并基于相邻轨迹点计算分段速度，若某个分段速度超过上限，也会判定记录异常。最后，当运行来源被标记为模拟模式且当前环境不允许模拟跑步时，系统拒绝提交结果。

以上规则并不能完全覆盖所有作弊场景，但对于课程设计和基本业务系统而言，已经形成了多维度的有效性校验框架。相较于只检查距离或时长的简单方式，该实现更符合校园跑业务对记录真实性的要求。

#### 5.3.5 跑步结果与历史设计
跑步通过校验后，后端会写入最终结果，并计算平均配速、卡路里、步数估算、运动置信度等信息。随后系统将该次跑步纳入个人累计里程和累计时长统计，并驱动后续打卡、计划推进、积分奖励和排行榜刷新。小程序端完成提交后会进入结果页，展示本次跑步的关键指标和路线地图。

用户在个人中心可以查看最近完成的几次跑步，在跑步历史页或结果页中还能查看单次详细记录。通过历史记录，学生能够对自己的运动强度和稳定性形成更直观的认知。

### 5.4 打卡、计划与成长模块设计与实现

#### 5.4.1 每日打卡逻辑
系统中的打卡不是独立按钮式签到，而是与有效跑步结果关联。用户完成一次通过校验的跑步后，后端会调用打卡服务，检查当天是否已存在打卡记录。若当天尚未打卡，则创建新的打卡记录，并根据前一天是否连续完成打卡自动计算连续天数；若当天已存在打卡记录，但新的跑步又对应某个计划日，则系统会补充计划关联信息。

这种设计能有效避免“空打卡”或“无跑步签到”的问题，使打卡更贴近实际运动行为。

#### 5.4.2 跑步计划逻辑
成长中心为学生提供三种计划模板，分别是 5km 入门、减脂跑和周目标计划。学生选择新的计划时，系统会自动取消原有激活计划，并创建新的用户计划与对应的逐日计划项。不同模板对应的计划日目标距离和目标时长通过服务端规则动态生成。

当用户完成一次有效跑步后，系统会自动对当前激活计划进行推进，寻找尚未完成的最早计划日，判断本次跑步是否达到目标距离或目标时长。如果达到，则该计划日标记为完成，并将当前计划推进到下一日；若所有计划日都已完成，则整个计划状态改为已完成。通过这种方式，跑步计划与真实运动结果形成自动联动，避免学生额外执行重复操作。

#### 5.4.3 积分、任务与等级逻辑
系统将成长反馈拆分为积分、任务和等级三个层次。用户每完成一次有效跑步，首先按照距离发放基础积分，规则为 `max(10, 距离 × 20)`。随后系统遍历所有激活任务，根据任务目标类型分别累加距离、跑步次数或时长。如果任务首次达到目标值，则将任务标记为完成，并额外发放任务奖励积分。

在等级方面，系统根据当前总积分匹配等级规则表中的阈值，自动更新用户等级值。由于积分既来源于跑步基础奖励，也来源于任务奖励，因此等级提升能够较好反映学生一段时间内的综合活跃度。

#### 5.4.4 徽章授予逻辑
徽章用于体现阶段性成就。系统支持按累计里程、跑步次数、积分、连续打卡、计划完成数和跑团参与情况等条件授予徽章。每次跑步后，服务端会重新检查用户是否达到了某个徽章的授予条件；若满足条件且此前未获得，则生成新的用户徽章记录。

这种设计让成长反馈不仅停留在数值层面，也具备了更强的可视化和仪式感。学生在成长中心和个人中心都能看到自己已获得的徽章，从而提升平台使用成就感。

### 5.5 社交模块设计与实现

#### 5.5.1 动态发布与浏览
系统提供动态广场，用于展示学生发布的运动内容。学生可以在动态页输入文本并发起发布请求，后端会创建动态记录，同时基于预设敏感词词典执行初步风险标签识别。如果正文中命中了敏感词，则该动态默认进入待审核状态；否则直接视为已通过审核。这样既兼顾了发布效率，也保留了必要的内容治理能力。

首页中的热门动态和动态页中的列表数据，均来自已审核通过的动态内容。动态对象中包含正文、图片链接、点赞数、评论数、举报数、风险标签、是否置顶和发布时间等信息。

#### 5.5.2 点赞、评论与举报
在互动层面，系统支持学生对动态点赞、评论和举报。点赞采用用户与动态唯一约束控制，同一用户不能重复点赞。评论以时间升序展示，便于阅读交流。举报功能则将举报原因写入专门数据表，同时增加动态的举报计数。当同一动态举报数达到一定阈值后，系统会自动把其状态改回待审核，交由后台进一步处理。

这一机制说明系统不仅鼓励互动，也考虑了校园内容管理的边界问题。

#### 5.5.3 跑团加入与群消息
跑团是本系统社交模块中较有校园场景特色的部分。学生可以查看当前活跃跑团列表，并选择加入某个跑团。加入后，系统会创建或激活该学生与跑团的成员关系，并刷新跑团人数。退出跑团时，系统不会物理删除成员关系，而是把活跃状态置为失效，便于保留历史数据。

只有已加入跑团的学生才能查看该跑团的成员和消息，也才能发送群消息。发送消息后，系统记录发送时间、发送人和消息内容，其他成员进入对应跑团详情页时即可查看最新交流内容。跑团模块增强了校园跑中常见的约跑、互相鼓励和信息通知场景。

### 5.6 活动与教程模块设计与实现
活动模块面向校园跑实际运营需求设计。管理员在后台填写活动标题、地点、开始与结束时间、报名截止时间、封面、描述和人数上限后，即可发布活动。学生端活动页和首页推荐区域会显示活动列表，学生可以查看详情并发起报名。报名时，后端会依次校验活动是否存在、状态是否为已发布、报名是否已截止、是否已重复报名以及人数是否已满，只有全部通过后才会生成报名记录。

教程模块用于发布跑步方法和训练建议。管理员可以在后台录入教程标题、摘要、正文和封面，小程序端则在活动页和首页展示最近发布的教程内容。教程虽然不直接参与校园跑成绩计算，但有助于提高系统的内容价值和使用黏性。

### 5.7 管理后台模块设计与实现

#### 5.7.1 运营概览
后台首页通过 `/api/admin/dashboard` 接口获取学生总数、累计里程、累计积分、待审核动态数量和最近操作日志等信息。页面采用卡片方式展示关键指标，并在侧边展示后台定位说明和最近日志摘要，帮助管理员快速了解系统整体情况。

#### 5.7.2 学生管理与导入
学生管理页支持查看学生列表和学生详情。详情信息包括学号、学院、班级、累计里程、累计时长、积分、等级和个人简介等内容。管理员可直接对学生状态进行启用或禁用操作。

更重要的是，后台提供了批量导入学生功能。系统支持 `.csv` 和 `.xlsx` 两类文件格式，推荐字段包括学号、姓名、学院、班级、用户名、密码和状态。导入时，系统会自动判断是新建学生还是更新已存在学生，并对必填项和状态值进行校验。对于新导入学生，如果没有提供初始密码，则系统会拒绝导入。该功能对真实校园环境中的批量建档具有较强实用意义。

#### 5.7.3 内容审核
后台审核页用于处理待审核动态。管理员可以查看动态内容、风险标签和举报数量，并执行通过、拒绝、下架和置顶四类操作。不同处理动作对应动态状态的不同变化，其中置顶动作会把动态标记为推荐内容，有利于首页或列表优先展示。审核完成后，系统会将动作写入审计日志。

#### 5.7.4 徽章与奖励维护
后台奖励页允许管理员新增徽章规则，包括徽章编码、名称、说明、图标、规则类型和触发阈值。规则类型覆盖累计里程、跑步次数、积分、连续打卡、计划完成和跑团参与等多个维度。该设计体现出成长激励体系并不是写死在代码中的固定展示，而是具备一定的后台配置能力。

#### 5.7.5 活动与教程发布
后台活动页同时支持活动发布和教程发布。活动表单用于录入时间和人数信息，教程表单用于录入摘要和正文信息。页面下方还会展示当前已发布活动和教程列表，方便管理员确认发布效果。

#### 5.7.6 数据看板与操作日志
后台统计页通过概览接口和趋势接口获取运行统计数据，包括当日活跃跑者数、当日里程、平均配速、打卡人数、学生总数、跑团成员数以及近 14 天趋势数据。页面使用折线图、柱状图和环形图进行展示，直观反映系统活跃情况和连续打卡分布。后台日志页则以表格形式展示关键操作时间、操作类型、目标对象和具体细节，用于日常追踪和问题定位。

### 5.8 关键业务流程说明

#### 5.8.1 学生跑步并触发成长反馈流程
学生登录成功后进入跑步页，点击开始跑步创建跑步会话。跑步过程中，小程序持续采集定位和传感器数据。用户结束跑步后，前端将距离、时长、步数、轨迹点和传感器摘要提交至后端。后端通过有效性校验后保存跑步结果，并依次更新用户累计数据、推进跑步计划、记录当日打卡、发放积分、更新等级、推进任务进度、授予徽章、清除排行榜缓存和刷新当日统计数据。随后学生在结果页、成长中心和个人中心都能看到对应反馈。

#### 5.8.2 动态从发布到审核的流程
学生发布动态时，后端先进行敏感词初筛。若未命中风险词，则动态直接处于通过状态；若命中风险词，则动态进入待审核状态。其他学生浏览已审核动态时，可进行点赞、评论和举报。若举报数量增加到一定阈值，系统会再次将动态送回待审核。管理员在后台审核页处理后，动态根据处理结果决定是否继续公开显示。

#### 5.8.3 活动报名流程
管理员先在后台发布活动，活动会同步出现在小程序首页推荐区域和活动页。学生浏览到感兴趣活动后进入详情页点击报名，后端依次校验活动状态、报名截止时间、是否重复报名和人数限制，校验通过则生成报名记录。该流程较为完整地覆盖了校园活动报名的基本约束。

### 5.9 本章小结
本章围绕系统真实代码实现，对登录认证、首页、跑步记录、打卡计划、成长激励、社交互动、活动教程和后台管理等核心模块进行了详细说明。通过这些模块协同，系统完成了从学生运动行为采集到后台运营支撑的完整闭环。

## 第六章 系统测试

### 6.1 测试目的
系统测试的目标是验证各模块是否能够按照预期协同工作，并检查关键业务规则是否能够正确执行。对于本文项目而言，测试重点不只是界面是否能打开，更重要的是跑步结束时的有效性校验、每日打卡与计划推进是否正确触发、动态审核链路是否闭环、后台导入和统计是否可用。

### 6.2 测试环境
本系统的测试环境采用项目提供的本地联调方案。服务端运行在 Spring Boot 环境下，数据库使用 MySQL 8.0，缓存使用 Redis 7，管理后台通过 Vite 构建后由浏览器访问，小程序端在微信开发者工具中运行。通过 Docker Compose 可以同时拉起相关容器，使测试环境尽量接近统一的开发部署环境。

### 6.3 功能测试

#### 6.3.1 登录与鉴权测试
测试内容包括学生账号登录、管理员账号登录、错误密码提示、禁用账号拦截和未授权访问跳转等。测试结果表明，系统能够在登录成功后正确签发令牌，并在令牌缺失或无效时拒绝访问受保护接口。管理后台在收到未授权响应后能够自动清理本地会话并回到登录页。

#### 6.3.2 跑步记录测试
跑步模块测试重点包括开始跑步、暂停恢复、结束提交、恢复未完成记录、放弃当前记录和结果页查看。测试中，系统能够正常记录距离、时长、配速和历史结果，并能在页面重进后恢复未完成状态。对于结束请求中时长过短、里程过短、轨迹点过少或速度异常的情况，系统能够返回明确错误提示并拒绝保存结果。

#### 6.3.3 打卡与计划测试
在完成有效跑步后，系统能够自动生成当日打卡，并在连续多日使用时更新连续天数。用户选定跑步计划后，符合目标条件的跑步记录能够自动推进计划日。若所有计划日均完成，计划状态会正确切换为已完成。

#### 6.3.4 成长激励测试
测试中，用户完成跑步后能够正确增加积分，并根据总积分变化更新等级。对于已配置任务，系统能够按目标类型累加任务进度；当进度达到阈值时，系统会标记任务完成并发放额外积分。对于已满足阈值的徽章，系统能自动授予且不会重复发放。

#### 6.3.5 社交互动测试
学生可正常发布动态、查看动态、点赞、评论和举报。重复点赞会被系统拒绝，举报达到阈值后动态会进入待审核状态。对于跑团功能，未加入成员无法直接查看跑团消息；加入后即可查看成员列表和发送消息，退出后消息访问会再次被限制。

#### 6.3.6 活动与后台测试
活动模块中，学生能够浏览活动并报名，系统会对重复报名、截止后报名和人数已满等情况做出限制。后台学生导入功能能够识别 `.csv` 和 `.xlsx` 文件并完成新增或更新；动态审核页可正确执行通过、拒绝、下架和置顶操作；徽章维护、活动发布、数据看板和日志展示均能正常运行。

### 6.4 单元测试分析
项目后端已经编写了多组单元测试，覆盖认证、跑步校验、签到、社交、计划、用户和后台服务等关键业务模块。以跑步校验测试为例，测试验证了轨迹点数量不足时系统应抛出异常；社交服务测试验证了未加入跑团的用户不能直接查看跑团消息；后台服务测试验证了审计日志字段为空时看板数据映射依然正确。这些测试说明项目在核心业务规则上已经具备一定的自动化验证基础。

### 6.5 测试结果总结
综合来看，系统已较好实现了预定的主要功能。学生端能够完成从登录、跑步、打卡、成长到社交的完整使用流程；后台能够完成从学生管理、内容审核、活动发布到数据分析的基本运营流程。虽然当前测试更多集中在功能正确性和核心规则层面，尚未覆盖更大规模并发和更复杂异常场景，但对于毕业设计初稿和本地演示系统而言，已经具备较好的完整性和可验证性。

## 第七章 总结与展望

### 7.1 总结
本文围绕高校校园跑场景，设计并实现了一套基于微信小程序的校园跑步打卡与社交系统。系统采用微信小程序、Spring Boot、Vue 3、MySQL、Redis 和 Docker Compose 等技术构建，完成了学生端、服务端和管理端的三端协同。与单纯记录跑步里程的小型应用相比，本文项目更强调完整业务链路，既覆盖了跑步开始、结束、历史查看等基础功能，也覆盖了打卡、计划、任务、徽章、排行榜、动态、跑团、活动和后台管理等扩展能力。

在实现过程中，本文重点解决了三类问题。第一类是跑步记录可信度问题。系统通过最小里程、最小时长、最少轨迹点和最大速度阈值等规则，提高了记录有效性。第二类是学生持续参与问题。系统把跑步结果与积分、等级、任务、徽章、连续打卡和计划推进进行联动，增强了长期使用动力。第三类是平台运营支撑问题。后台系统实现了学生导入、内容审核、活动发布、数据看板和审计日志等功能，使平台具备一定的管理能力。

总体来看，本文项目已经实现了校园跑平台的核心闭环，具备较好的功能完整性和实践价值。对于毕业设计而言，该系统不仅体现了前后端分离、权限控制、数据库设计、缓存使用和多模块协同的工程实现能力，也对校园跑数字化管理提供了具有操作性的参考方案。

### 7.2 展望
尽管系统已经完成了主要功能，但仍有若干方面值得继续完善。

首先，跑步异常识别规则仍可进一步增强。当前实现主要依赖阈值校验和分段速度检查，后续可引入更细粒度的轨迹分析、定位漂移修正和多传感器联合判定机制，提高反作弊能力。

其次，排行榜和统计维度仍较基础。目前系统以总里程排行榜为主，未来可以增加按学院、班级、周维度和活动维度的多层级排行，并支持更灵活的数据筛选。

再次，社交功能仍有扩展空间。未来可增加图片上传、@提醒、消息通知、活动群聊和更丰富的互动反馈，使平台更具社区氛围。

最后，后台运营能力仍可继续深化。当前后台已经具备基础数据看板，但还可以进一步加入异常用户监测、活动成效评估、内容热度分析和规则参数化配置等能力，使系统从“可用”走向“更好用”。
